Authentication

eFFECT uses two different authentication mechanisms depending on the environment: JWT for development and testing, and Azure Entra ID for production. This allows for flexibility in development while ensuring secure, role-based access in production.

Production Authentication (Azure Entra ID)

In the production environment, eFFECT uses Azure Entra ID (formerly Azure Active Directory) for secure authentication and role-based access control.

This setup ensures enterprise-grade identity protection and centralized user management for FFT teams.

🔐 Login Flow​

  1. The user accesses the eFFECT production app.
  2. The app redirects the user to the Microsoft login portal.
  3. After successful login, Azure Entra ID returns an access token.
  4. The backend validates the token using Microsoft's public keys.
  5. The app identifies the user and role.

⚙️ Azure Configuration Requirements​

To enable Azure Entra ID authentication, the following resources must be configured in the Azure Portal:

  • App Registration
    • Redirect URIs
    • Token configuration (including group claims)
  • Client Secret
  • Admin & User Groups

⚠️ Environment Configuration​

Secure Secrets Management

The frontend and backend rely on environment variables for credentials and other sensitive configuration. These are securely stored in GitHub Secrets and automatically injected during deployment via GitHub Actions.

Do not commit sensitive variables to the repository.

🔐 Development & Test Authentication (JWT)​

In dev and test environments, authentication is based on JSON Web Tokens (JWT).

Login Flow​

  1. A user logs in via a custom login form.
  2. The backend validates the credentials (usually a mock or test user base).
  3. A JWT is generated and sent to the frontend.
  4. The frontend stores the token.

JWT Details​

  • The JWT contains userID and role.
  • The backend verifies the token on every protected route.

Environment Configuration​

Ensure the .env file includes:

JWT_SECRET_KEY=your_dev_or_test_secret
